Hyperion started as a set of Rust microservices - a user service, a subscription
service, and an AI service, each behind its own backend-for-frontend gateway -
built deliberately as a learning project in microservices system design, not to
ship a product.

That shape was fine for learning, but the operational cost (Kafka wiring,
per-service deploys, duplicated auth/DB-pool code) wasn't worth it once the
goal changed to something real and maintainable long-term. Rather than
running several small services I could barely keep up with alone, I moved
everything into one Cargo workspace: one deployable per app, compiler-enforced
boundaries between domain crates, and a shared base - auth, sessions, DB pool,
email, AI - poured once instead of copied per service.

Moving to a monorepo didn't mean giving up modularity - it's still the whole
point. Domain crates stay decoupled from each other by the compiler, not by
convention, so any of them could be pulled back out into its own service later
if a real scaling need ever shows up. The workspace's own docs are built the
same way: a Next.js docs app renders the crate and architecture docs (with the
same diagram pipeline used here) as a single browsable book, kept next to the
code instead of drifting in a wiki.
