Most rendering bugs I've chased came from the same root cause: a component that
knew too much. It held the data shape, decided the colour, picked the icon, and
drew the markup — all in one place. Change any one of those and you risked the
other three. The seam pattern is the discipline of pulling those concerns apart
so each lives in exactly one layer.

## Three layers, one concept

A domain concept flows through three modules, each with a single job:

- **Data** owns the shape and the values — no JSX, no colours, no icons.
- **A presentation seam** resolves that data into something visual — an icon, a
  hue, a piece of markup.
- **A component** consumes the seam's output and never reaches past it.

The rule that keeps this honest is exhaustiveness. When the seam maps a closed
set of names to colours, a missing entry is a compile error, not a silent gap:

```ts
const CATEGORY_HUES: Record<CategoryName, BadgeCategory> = {
  Engineering: "indigo",
  Workflow: "green",
  // omit a vocabulary name here and `tsc` fails — the gap can't reach runtime
};
```

### Why the compiler, not a test

A unit test proves the mapping is right *today*. The exhaustive `Record` proves
it stays right *tomorrow* — the moment someone adds a category, the build stops
until they resolve its hue. That's the difference between catching a regression
and preventing one.

## What it buys you

Adding a category becomes a two-file edit with no threading through call sites.
Auditing "where do colours come from" becomes a one-file read. And an agent
editing the codebase has a narrow, typed surface to work against instead of a
component that mixes four concerns. The pattern is documented further in the
[Next.js App Router guide](https://nextjs.org/docs/app), which leans on the same
separation between data loading and presentation.

The cost is one extra module per concept. In return, every change touches one
layer at a time — which is the whole point.
